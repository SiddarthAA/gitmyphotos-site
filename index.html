[02] THE PROBLEM WITH CLOUD

The Subscription Trap Has An Expiry Date On Your Memories.

/ THE TRAP /

Every major photo platform runs the same playbook. Draw you in with free storage. Compress your originals. Lock your library behind proprietary formats. Then raise the price and dare you to leave. Migrating away costs weeks of your life — if the export tool works at all. Your photos aren't stored. They're held.

/ THE EXIT /

GitMyPhotos is not a storage provider. It is a pipeline. Your photos are pushed, in full resolution, to a private GitHub repository that belongs entirely to you. We run locally on your machine. We touch nothing we don't need to. The day you stop using GitMyPhotos, every single photo is still exactly where you left it — in your repo, yours forever, no extraction required.

[03] THE ENGINEERING

One Upload.
Six Steps.
One Commit.

/INGESTION/

Drop a photo and the pipeline fires immediately.

EXIF data extracted. GPS coordinates resolved. Collision-proof filename stamped from capture time. Thumbnail generated at 400px. Metadata sidecar written. Everything batched into a single atomic Git operation. Ten photos or ten thousand — always one commit.

/STRUCTURE/

A Repository That Thinks For You.

Three folders. One manifest. Zero ambiguity. originals/ holds full-resolution files, untouched. thumbs/ holds fast-loading previews. meta/ holds a JSON sidecar per photo. manifest.json indexes everything. Date-partitioned. Human-readable. Git-native.

/RETRIEVAL/

Near-Zero GitHub API Usage At Browse Time.

The manifest loads once per session — one API call. Thumbnails stream from local disk after the first view. Previews generated on demand, cached immediately. Originals fetched directly from your repo on request. A library of ten thousand photos costs the same API budget as a library of ten. One call. That's the ceiling.

[05] THE PIPELINE

How a photo goes from your device
to your repository in under ten seconds.

Four stages.
One atomic commit.
Zero compromise.

01
RECEIVE
/ STEP 01: INGEST

File lands in the ingest layer.

Format validated. Size checked. A collision-proof ID is stamped using the EXIF capture timestamp down to the millisecond.

02
PROCESS
/ STEP 02: PROCESS

EXIF pulled. GPS resolved. Thumbnail generated.

Metadata JSON compiled. Three artefacts — original, thumb, sidecar — staged and ready for a single atomic operation.

03
COMMIT
/ STEP 03: COMMIT

One Git Tree API call. Atomic. Permanent.

Original, thumbnail, metadata sidecar, updated manifest and regenerated README — committed together. Verifiable. Irreversible.

04
SERVE
/ STEP 04: SERVE

Manifest cached. Thumbnails from disk. API: zero.

Previews generated once, cached forever. Originals pulled direct from your repo on request. GitHub API usage: effectively zero.

[06] GETTING STARTED

Three steps.
One command.
Operational in under five minutes.

01
/01 ─── DEPLOY

Clone & Boot

git clone the repository. Copy .env.example to .env. Run docker compose up. Two containers start: frontend on :3000, backend on :8000. No cloud accounts. No API keys. No configuration beyond a single file.

02
/02 ─── AUTHORISE

Connect Your Repository

Open localhost:3000. Authenticate via GitHub OAuth — one click. Choose an existing private repository or provision a new one from the UI. GitMyPhotos scaffolds the folder structure and commits the initial manifest in one atomic operation. You're live.

03
/03 ─── OPERATE

Upload. Commit. Own.

Drag photos into the interface. The pipeline runs on your machine. Every photo lands in your repository. Browse through the local cache layer. Pull up a single photo. Download the original at any time. Nothing is stored anywhere but your repo.
